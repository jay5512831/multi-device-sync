# 架构 B：直接同步模式

适用于**支持**同步点号文件夹的服务（OneDrive、Dropbox、Google Drive、Syncthing、WebDAV、rsync）。

## 工作原理

`.workbuddy/` 直接被云服务同步，不需要中转目录。代价是：如果两台设备没有正确切出/切入就同时编辑，会产生冲突副本。

## 锁文件机制

`.workbuddy/` 中的 `.sync-lock` 文件作为软锁来协调设备。

> **完整锁文件逻辑（4种场景）**定义在 `SKILL.md` → "锁文件逻辑"章节。
> 本文档只覆盖要点。

### 锁文件格式
```yaml
device: "办公室"
timestamp: "2026-04-20T18:00:00+08:00"
```

- **切入时**：检查锁 → 如果是残留/过期锁则清理；如果是其他设备的新锁则短暂等待
- **切出时**：写入锁，包含当前设备 ID + 时间戳

## 冲突检测

切入时，在执行其他操作之前先扫描 `.workbuddy/` 中的冲突副本。


完整正则列表见 `references/conflict-patterns.md`。

### 解决流程

发现冲突时：
1. 列出所有冲突文件及其原始对应文件
2. 对每一对：
   a. 显示文件大小和修改日期
   b. 如果是文本文件：显示 diff
   c. 询问用户："保留原文件 / 保留冲突副本 / 查看完整差异？"
3. 删除已解决的冲突副本
4. 在记忆日志中记录解决情况

## 冲突预防最佳实践

1. **离开设备前一定要切出** —— 这是防止冲突的第一措施
2. **设置每日提醒**，如果你容易忘记（技能会作为可选引导提出）。注意：定时任务每次触发会创建新对话
3. **不要同时在两台设备上编辑** —— 此模式设计用于顺序使用，不是并发编辑
4. **检查同步状态** —— 切入前确认云服务已完成同步（查看系统托盘图标）

## rsync 特别说明

rsync 通常是单向的。用于多设备同步时：
- 方案一：使用中央服务器作为枢纽，每台设备从中推送/拉取
- 方案二：rsync 只用于切出（推送），依赖服务器完成切入（拉取）
- 无内置冲突检测——锁文件机制在此架构下尤为重要
